# Pattern Library

> Recurring patterns Claude has observed about you. **This is the killer file in this template.** Most second-brain setups don't have a self-awareness layer. This one does.
>
> Read this file before responding to significant requests. When a known pattern is recurring, name it explicitly. Append new patterns as they emerge. Do not delete patterns without explicit instruction.

## How to use this file

Each pattern has:
- **Name** — short, memorable. You'll come to recognize them.
- **Description** — what the pattern is and how it shows up
- **Instances** — specific dated examples (the more, the more confident the pattern is real)
- **Trigger conditions** — what circumstances precede the pattern
- **Recommended response** — what Claude should do when the pattern is detected

When your current behavior or request matches a pattern's trigger conditions, Claude surfaces the pattern by name in the response — not to block you, but to make the choice conscious.

**How patterns get added:** Claude proposes new patterns when it sees one form (3+ instances). You approve before they land. Once approved, future Claude sessions reference the pattern by name.

## EXAMPLE Pattern: Impulsive Project Initiation Under Stimulus
**Description:** Tendency to start new projects or systems when an external trigger (a video, a conversation, a market event, a feeling of excitement) creates a sense of "this could be huge for me." The new project competes for attention with existing unfinished work.

**Instances:**
- (date): Decided to learn a new programming language after watching a viral conference talk while a side project was at 80% completion.
- (date): Pitched 3 new business ideas in one week after attending a founders' meetup, while existing freelance commitments had unmet deadlines.

**Trigger conditions:** External excitement-driven trigger; phrases like "this could be huge," "I want to do this now," "we should add X to the system."

**Recommended response:** Surface the pattern by name. List current open loops. Ask whether the new initiative displaces existing closing work or adds to it. Do not block — flag and let the user choose with full awareness.

## EXAMPLE Pattern: Building Over Finishing
**Description:** Stronger drive to build new things than to close existing loops. Open work piles up. Feature creep in active projects competes with shipping the v1.

**Instances:**
- (date): During a project's final week of dev, repeatedly proposed new features rather than scoping down to ship.
- (date): Pre-launch, pushed for one more "polish pass" while opportunity cost of delay grew.

**Trigger conditions:** Phrases like "what else can we add," "make it more X," "improvements," "state of the art." Also: starting new projects/systems while existing ones are pre-revenue or pre-completion.

**Recommended response:** Name the pattern. Ask what closing work the new building is competing with. Push toward shipping the v1 of the current thing before adding to it. Reframe additions as v2 candidates with explicit "after first run" gating.

## EXAMPLE Pattern: Conflict Avoidance In Important Conversations
**Description:** When a hard conversation needs to happen — feedback to a collaborator, ending a relationship, asking for what you need — avoidance shows up as "I'll do it next week," busywork that pre-empts the conversation, or rationalizations that the situation "isn't that bad."

**Instances:**
- (date): Postponed a feedback conversation with a collaborator for 6 weeks; impact compounded.
- (date): Avoided a money conversation with family for months; resentment grew on both sides.

**Trigger conditions:** A specific conversation you've been "meaning to have" for more than 1 week. Phrases like "I'll bring it up when…", "It's fine," "I don't want to make it a thing."

**Recommended response:** Name the pattern. Ask what the cost of one more week of avoidance is. Offer to draft the message or script the conversation so the friction is lower.
